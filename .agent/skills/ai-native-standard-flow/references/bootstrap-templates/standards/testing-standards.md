# Testing Standards

> 模板提示：本文仅供参考，请结合项目实际进行人工完善与确认。

## 测试分层

| 层级 | 目标 | 工具示例 | 运行时机 |
|------|------|---------|---------|
| 单元测试（Unit） | 验证单个函数/模块逻辑 | Jest / Vitest / pytest | 每次提交（本地 + CI） |
| 集成测试（Integration） | 验证模块间交互与数据流 | Supertest / httpx | PR 合并前（CI） |
| E2E 测试 | 验证完整用户路径 | Playwright / Cypress | 发布前（CI） |

## 覆盖率基线

<!-- 根据项目实际设置，建议值仅供参考 -->

| 层级 | 覆盖率目标 | 强制/建议 |
|------|----------|---------|
| 单元测试 | ≥ 80%（行覆盖） | 强制 |
| 集成测试 | 覆盖所有公开 API | 强制 |
| E2E 测试 | 覆盖核心用户路径 | 建议 |

## 关键路径优先级规则

- **支付/权限/数据一致性**相关变更：单测 + 集成测试双覆盖，且需人工拍板。
- **新功能**：至少包含单测，集成测试视接口复杂度决定。
- **Bug 修复**：必须附回归测试，防止相同 bug 复现。

## 使用规范

- 每个任务至少绑定一项可执行验证（验收标准驱动测试）。
- 先本地验证通过，再提交 PR 触发 CI。
- 测试文件与被测模块同目录或统一放 `tests/` 目录，保持一致。
- Mock 外部依赖（数据库、HTTP 请求），单测不依赖网络或环境。
