# 团队规则（审核对照用）

> **用法**：助手按 **① §A 结构与职责（硬闸）→ ② §B 迭代与质量（过程）→ ③ 发现/注册与变更记录** 对照；依据写明命中条款或「不适用」及原因。**结论须可追溯到下列条款**。  
> 与他处文档口径冲突时：**以本文件为准**；先更新本 reference，再同步各 skill。

---

## 审核标准

### A. 结构与职责（硬闸）

#### A.1 本质不变

- **定义**：skill 的「本质」= `description`（含 WHAT/WHEN）与 `SKILL.md` 正文承诺的**核心任务与边界**。
- **审核**：是否**改本质**（任务类型漂移、边界无限扩大）。若是 → 不通过或附条件（拆 skill、改 `name`/目录、重写 `description` 与正文边界，并 sync / changelog）。
- **允许**：同本质下的措辞、步骤细化、示例、bug 修复。

#### A.2 单一职责

- **审核**：是否把**另一套主任务**硬塞进本 skill。若是 → 拆出或迁回既有 skill。
- **关系**：skill 之间**引用/分工**优于「一包到底」。

#### A.3 主流程与可选项（多数 vs 小概率）

- **审核**：小概率场景是否被写进**必做主流程**。若是 → 不通过或附条件（改为「主流程结束后的可选项」+ 用户确认）。
- **落地建议**：主流程标明默认完成点；**可选项 / 进阶**单列；助手在**优化建议**中可给出拆分方案。
- **优化职责**：**先**出审核结论与依据，**再**给优化方案；不得以「不通过」为唯一输出而不给可落地改法（本质/单一职责硬伤除外）。

### B. 迭代与质量（过程）

#### B.1 审核时要问的核心问题

| 维度 | 审核时看什么 |
|------|----------------|
| 价值 | 改动是否会**降低**用户目标完成度？若会，不通过或附条件。 |
| 范围 | 是否**一事一改**、无无关重构？（与「最小必要改动」、§A.2 一起看） |
| 证据 | 新删改规则是否有**可核对依据**（反馈、问题、提交、样例）？无依据则附条件或要求补证据。 |
| 自检 | 改规则处是否**同步**提到/更新了可验收的自检或等价表述？**助手只判定缺不缺、不代写全文**。 |
| 兼容 | 是否不必要地打破既有稳定用法？能兼容则优先指出兼容写法。 |
| 事实 | 表达可优化，**事实与边界**不可在无证据下被改掉。 |
| 可追溯 | 能否一句话说清「为何改、改了什么、影响谁」？changelog 是否将补这一条？ |
| 场景 | 规则是否服务**高频真实场景**？与 **§A.3** 一起看，避免为「完整」拖死主流程。 |

#### B.2 变更前门槛（拟议是否配进入实现）

以下四条**未满足则倾向附条件通过**（先补全再改 skill）：

1. 目标能否**一句话**说清；2. 依据是否**可核对**；3. 影响范围是否点到（章节/模板/自检/示例）；4. 成功标准是否明确。

#### B.3 变更后验收（依据中逐条表态）

对本次实际改动，在审核**依据**里对下列各项 **满足 / 不满足 / 不适用**（不满足则进结论或附条件）：

- [ ] 规则与自检（或可验收表述）**同步**
- [ ] 无与目标无关的结构大改
- [ ] 示例/模板**无事实漂移**
- [ ] 版本记录将含「背景 / 动作 / 结果 / 影响」
- [ ] 读者无需口头补充即可理解本次变更意图

#### B.4 与各 skill 文档的分工

- **本 reference**：定义「审什么、按什么标准通过」。
- **各 skill 目录** `changelog.md` / `版本变更.md`：只记「**该 skill** 发生了什么变更」。

## 变更记录

- 有实质约定或行为变更时，在本 skill 目录 `changelog.md` 追加条目；其它 skill 从其目录约定文件名（如 `版本变更.md`）。
