## ADDED Requirements

### Requirement: 任务页遵循原型视觉结构并展示动态长期任务分区
系统 MUST 在任务页基于真实数据展示长期任务分区，包括当前专注任务、暂停的任务库与已达成的里程碑；系统 MUST 承袭 `docs/prototype/v1.0/screens/ios-task-page.html` 的视觉结构、信息层级与组件组织，而不是使用默认系统列表堆叠替代。系统 MUST 在顶部栏下方、当前 TopOne Hero 上方展示任务页精神性导语，但不得让导语削弱当前 TopOne 作为任务内容第一视觉锚点的地位。

#### Scenario: 展示顶部任务页导语
- **WHEN** 用户进入任务页
- **THEN** 系统在顶部栏下方展示主标题与副标题组成的导语段落
- **THEN** 导语用于表达产品精神性与专注方向
- **THEN** 当前 TopOne Hero 仍是任务内容区域最强的视觉锚点

#### Scenario: 存在当前 TopOne
- **WHEN** 用户已经设置当前 TopOne
- **THEN** 任务页以原型中的深色 Hero 卡片作为第一视觉锚点展示当前 TopOne
- **THEN** 卡片展示当前 TopOne 的标题、手动百分比与剩余锁定时间倒计时
- **THEN** 百分比使用强调色视觉层级，并支持点击编辑与滑动调整
- **THEN** 展示内容来自持久化任务状态，而不是原型静态示例文案

#### Scenario: 存在暂停长期任务
- **WHEN** 用户存在未完成且非当前 TopOne 的长期任务
- **THEN** 任务页在暂停的任务库中展示这些任务
- **THEN** 暂停任务按 `S / A / B / C` 分组
- **THEN** 暂停任务默认折叠或弱化显示
- **THEN** 暂停任务区视觉强度 MUST 明显低于当前 TopOne Hero 卡片
- **THEN** 暂停长期任务条目不得使用强调型图标，避免与当前 TopOne 争夺视觉注意力

#### Scenario: 存在已完成长期任务
- **WHEN** 用户存在已完成长期任务
- **THEN** 任务页在已达成的里程碑区展示这些任务
- **THEN** 里程碑区默认折叠
- **THEN** 已完成长期任务不再占用当前 TopOne 区域

### Requirement: 长期任务支持增删改查
系统 MUST 支持创建、查看、编辑、删除长期任务，并保存标题、等级、手动进度、创建时间、锁定截止时间与完成时间字段；长期任务不得使用日常任务式通用状态字段。

#### Scenario: 创建长期任务
- **WHEN** 用户提交有效的长期任务标题与等级
- **THEN** 系统创建长期任务
- **THEN** 新长期任务默认进入暂停任务库，除非用户明确设为当前 TopOne

#### Scenario: 编辑长期任务
- **WHEN** 用户编辑长期任务标题或等级
- **THEN** 系统保存更新后的长期任务信息

#### Scenario: 删除长期任务
- **WHEN** 用户确认删除长期任务
- **THEN** 系统删除该长期任务
- **THEN** 系统同时删除该长期任务下的所有日常任务，避免留下孤立日常任务

#### Scenario: 限制长期任务数量
- **WHEN** 当前专注任务与暂停任务库中的长期任务总数已经达到 3 个
- **THEN** 系统拒绝继续创建新的未完成长期任务

### Requirement: 当前 TopOne 保持唯一并受锁定约束
系统 MUST 保证同一时刻最多只有一个长期任务处于当前 TopOne；如果已有当前 TopOne，用户必须先解绑当前 TopOne，才能选择其他长期任务作为新的 TopOne。

#### Scenario: 无当前 TopOne 时设置当前 TopOne
- **WHEN** 当前不存在 TopOne
- **AND** 用户将某个长期任务设为当前 TopOne
- **THEN** 系统要求用户选择锁定时间
- **THEN** 系统将该长期任务标记为当前 TopOne

#### Scenario: 已有当前 TopOne 时禁止直接切换
- **WHEN** 当前已经存在 TopOne
- **AND** 用户尝试将另一个长期任务设为 TopOne
- **THEN** 系统拒绝直接切换
- **THEN** 系统提示用户先解绑当前 TopOne

#### Scenario: 支持锁定时间选项
- **WHEN** 用户选择锁定时间
- **THEN** 系统 MUST 以 `ios-goal-commitment-lock.html` 的专注选择 / 锁定确认弹层为交互参考
- **THEN** 系统支持 `7 天`、`14 天`、`30 天` 与自定义时间
- **THEN** 自定义锁定跨度最长不超过 `180 天`
- **THEN** 系统保存当前 TopOne 的锁定截止时间
- **THEN** 展示层基于该时间计算并展示剩余锁定时间倒计时，而不是直接展示截止时间

#### Scenario: 锁定期内提前解绑
- **WHEN** 当前 TopOne 仍在锁定期内
- **AND** 用户尝试解绑当前 TopOne
- **THEN** 系统 MUST 进入参考 `ios-force-switch.html` 的强制更换页或全屏弹层
- **THEN** 系统要求用户填写提前更换理由
- **THEN** 页面实时展示理由字数差值，未达标为负数并使用克制错误色，达标显示 `0` 并使用成功色
- **THEN** 页面提供 `再坚持一下` 与 `确定放弃` 两类动作
- **THEN** 未达到最少字数门槛时系统拒绝解绑
- **THEN** 解绑完成后用户才能选择其他长期任务作为新的 TopOne

#### Scenario: 更换理由门槛变化
- **WHEN** 用户提前更换一次 TopOne
- **THEN** 系统记录一次提前更换次数
- **THEN** 下一次更换理由最少字数门槛按 `50 × 2^提前更换次数` 计算
- **THEN** 门槛不低于 50 且不高于 500

#### Scenario: 完成长期任务降低后续更换成本
- **WHEN** 当前 TopOne 的进度达到 100%
- **THEN** 系统将该长期任务标记为已完成
- **THEN** 系统解除该长期任务的当前 TopOne 状态与锁定截止时间
- **THEN** 系统将提前更换次数减半后用于后续门槛计算
- **THEN** 系统展示轻量完成庆祝反馈

### Requirement: TopOne 进度可手动维护
系统 MUST 支持维护当前 TopOne 的手动百分比进度，并将进度限制在 0 到 1 的范围内。

#### Scenario: 滑动更新进度
- **WHEN** 用户拖动当前 TopOne 卡片上的进度条
- **THEN** 系统更新当前 TopOne 的手动百分比进度

#### Scenario: 手动输入进度
- **WHEN** 用户点击当前 TopOne 卡片并输入百分比
- **THEN** 系统保存输入后的进度值

#### Scenario: 限制进度边界
- **WHEN** 用户提交小于 0 或大于 1 的进度值
- **THEN** 系统将保存值限制到 0 到 1 的范围内

### Requirement: 日常任务必须隶属于长期任务
系统 MUST 将日常任务归属到某个长期任务；创建时可选择任意长期任务，若存在当前 TopOne 则默认选中当前 TopOne。

#### Scenario: 创建日常任务时存在当前 TopOne
- **WHEN** 用户在任务页选择新增日常任务
- **AND** 当前存在 TopOne
- **THEN** 系统默认选择当前 TopOne 作为日常任务所属长期任务
- **THEN** 用户可以改选其他长期任务作为所属长期任务

#### Scenario: 创建日常任务时没有当前 TopOne
- **WHEN** 用户在任务页选择新增日常任务
- **AND** 当前没有 TopOne
- **THEN** 系统允许用户选择任意长期任务作为所属长期任务
- **THEN** 系统不得因为缺少当前 TopOne 而拒绝创建日常任务

#### Scenario: 查看暂停任务的日常任务
- **WHEN** 用户展开暂停任务库中的长期任务
- **THEN** 系统允许查看该长期任务下的日常任务

### Requirement: 日常任务支持增删改查与受限状态管理
系统 MUST 支持任意日常任务的创建、查看、编辑、删除；系统 MUST 只允许更新当前 TopOne 下日常任务的状态。

#### Scenario: 编辑日常任务
- **WHEN** 用户编辑任意日常任务的标题、等级或所属长期任务
- **THEN** 系统保存更新后的日常任务信息

#### Scenario: 删除日常任务
- **WHEN** 用户确认删除任意日常任务
- **THEN** 系统删除该日常任务

#### Scenario: 切换当前 TopOne 下的日常任务为执行中
- **WHEN** 用户将当前 TopOne 下的日常任务状态切换为 `执行中`
- **THEN** 系统记录开始时间

#### Scenario: 切换当前 TopOne 下的日常任务为已完成
- **WHEN** 用户将当前 TopOne 下的日常任务状态切换为 `已完成`
- **THEN** 系统记录结束时间

#### Scenario: 拒绝更新非当前 TopOne 下的日常任务状态
- **WHEN** 用户尝试更新非当前 TopOne 下的日常任务状态
- **THEN** 系统拒绝该状态更新

#### Scenario: 日常任务默认排序
- **WHEN** 系统展示当前 TopOne 的日常任务列表
- **THEN** 系统按 `执行中`、`未开始`、`已完成` 的顺序排序

### Requirement: 日常任务数量受限制
系统 MUST 按所属长期任务限制活跃日常任务数量，避免任一长期任务的任务池过载。

#### Scenario: 限制未完成任务总数
- **WHEN** 某个长期任务下 `未开始 + 执行中` 的日常任务数量已经达到 5 个
- **THEN** 系统拒绝继续为该长期任务创建新的未完成日常任务

#### Scenario: 限制执行中任务数量
- **WHEN** 某个当前 TopOne 下 `执行中` 的日常任务数量已经达到 2 个
- **AND** 用户尝试将另一个日常任务切换为 `执行中`
- **THEN** 系统拒绝该状态切换

### Requirement: 任务页提供动态操作入口
系统 MUST 在任务页提供添加、编辑、删除、状态更新与进度管理入口，且所有入口操作真实数据而非静态原型内容。

#### Scenario: 点击任务页添加按钮
- **WHEN** 用户点击任务页右下角 `+` 按钮
- **THEN** 系统 MUST 进入统一的创建任务页面，而不是先展示动作菜单
- **THEN** 创建任务页面 MUST 提供 `日常任务` 与 `长期任务` 类型切换
- **THEN** 创建任务页面 MUST 参考 `ios-goal-create.html` 的 `任务精炼` 结构、深色核心输入卡、次级字段卡片与底部名言区
- **THEN** `+` 按钮 MUST 保持右下角圆形浮动按钮形态，不得只放在导航栏工具按钮中
- **THEN** `+` 按钮 MUST 根据可用屏幕尺寸动态适配右侧与底部间距，并保持接近右下角的悬浮入口位置

#### Scenario: 左滑日常任务
- **WHEN** 用户左滑当前 TopOne 下的日常任务条目
- **THEN** 系统 MUST 参考 `ios-task-page-swipe-actions.html` 展示横向动作面板
- **THEN** 系统提供 `更新`、`编辑`、`删除` 入口
- **THEN** 仅当该日常任务属于当前 TopOne 时，系统提供或启用 `更新` 入口
- **THEN** 系统不得只使用行内 `Menu` 替代左滑动作

#### Scenario: 原型作为视觉与流程约束
- **WHEN** 实现任务页 UI
- **THEN** 系统 MUST 参考原型的信息架构、视觉层级、色彩方向、卡片圆角、浮动按钮、左滑动作与锁定 / 强制更换流程
- **THEN** 系统 MUST 保持动态数据驱动，不得依赖原型中的静态示例任务、示例标题或写死百分比作为正式内容
- **THEN** 系统不得把任务页实现成普通管理后台、系统默认表单堆叠或待办工具面板

#### Scenario: 底部导航提供后续页面占位
- **WHEN** 用户查看任务页根界面
- **THEN** 系统提供任务、奖励、设置三个底部导航入口
- **THEN** 任务入口展示当前任务页
- **THEN** 奖励与设置入口仅展示占位内容，不实现奖励池正式管理或设置偏好管理
- **THEN** 底部导航 MUST 贴合底部安全区，不遮挡任务页主要内容
