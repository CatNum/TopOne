## Context

`docs/requirements/` 已经确认任务页是当前 `TopOne` 的主要管理页面，必须始终突出当前唯一目标，而不是平铺所有目标。`docs/prototype/v1.0/screens/ios-task-page.html` 提供任务页的信息架构、视觉层级和关键交互参考，但本 change 的目标是实现可操作的动态程序，而不是复刻静态原型页面。

当前约束：
- 技术栈保持 `Swift 6` + `SwiftUI` + `SwiftData`。
- 当前优先平台为 `iOS`、`iPadOS`；验证链路仍可使用现有 `macOS` scheme 覆盖领域与服务规则。
- 日常任务必须隶属于长期目标；创建时可选择任意长期目标，默认选中当前 `TopOne`；当前没有 `TopOne` 时也允许创建日常任务。
- 本 change 要覆盖完整任务页闭环，包括锁定时间与锁定期内提前切换理由。
- 不在本 change 中实现奖励池正式管理、奖励抽取、完整完成归档成就页或 macOS 桌面常驻完整体验；允许接入奖励 / 设置底部导航占位页与轻量完成庆祝反馈。

## Goals / Non-Goals

**Goals:**
- 建立长期任务与日常任务的数据模型、等级与归属关系；长期任务不设置通用状态字段，通过当前 TopOne 绑定、完成时间与归属列表区分展示区域。
- 实现任务页动态展示：当前专注任务、日常任务、暂停的任务库、已达成的里程碑。
- 实现长期任务与日常任务的添加、编辑、删除。
- 实现当前 TopOne 的手动百分比进度管理。
- 实现日常任务状态管理，并按 `执行中`、`未开始`、`已完成` 默认排序。
- 实现 TopOne 锁定时间选择：`7 天`、`14 天`、`30 天`、`自定义`。
- 实现锁定期内提前切换时的强制更换理由输入与字数门槛规则。
- 用测试覆盖数量限制、唯一 TopOne、进度边界、日常任务状态排序、锁定与提前切换理由规则；UI 级验收以实现检查与构建验证记录为主，不引入独立快照测试。

**Non-Goals:**
- 不实现奖励池 CRUD。
- 不实现完成后的奖励抽取与成就档案绑定。
- 不实现完整完成归档页面；允许任务完成时展示轻量庆祝反馈，并将长期任务进入里程碑区。
- 不实现 macOS 桌面常驻层与状态栏交互。
- 不追求与 HTML 原型逐像素一致。

## Decisions

### 1. 任务页必须是动态程序，但视觉结构与关键流程要继承原型

选择：SwiftUI 实现必须围绕真实模型、查询结果、状态切换和用户动作构建，同时必须承袭 `docs/prototype/v1.0/screens/ios-task-page.html`、`ios-task-page-swipe-actions.html`、`ios-goal-commitment-lock.html` 与 `ios-force-switch.html` 的视觉结构、信息层级和关键交互。原型不是静态数据来源，但它是任务页视觉语言和流程组织的实现依据。当前实现还保留顶部精神性导语作为顶部栏下方、当前 TopOne Hero 上方的第一段引导，用于强化产品气质与页面进入感。

原因：任务页的核心价值是管理真实长期任务和日常任务；如果只做静态复刻，会阻碍业务规则落地。但如果只做系统默认控件堆叠，也会失去 TopOne 的产品气质、视觉主次和交互承诺感。正确做法是：数据和状态动态化，页面结构、组件形态、层级、按钮位置、左滑动作、锁定弹层和强制更换页按原型方向实现；顶部导语服务于精神性表达，但当前 TopOne Hero 仍应是任务内容本身的最强视觉锚点。

备选：按原型一比一还原静态视觉页面。该方案截图接近但不可用。另一个备选是仅用系统 Form/List/Menu 快速接功能，该方案可交互但不符合当前原型设计风格和产品体验要求。

### 2. 长期任务不设置通用状态字段

选择：长期任务模型不设置 `未开始` / `执行中` / `已完成` 这类通用状态字段；当前专注由唯一 TopOne 绑定表达，暂停任务库由“未完成且非当前 TopOne”的长期任务推导，已达成里程碑由完成时间或完成标记表达。

原因：长期任务不是日常任务，不需要日常任务式状态机；用状态字段混合表达当前专注、暂停、里程碑会让业务语义变模糊。

备选：为长期任务增加统一状态枚举。该方案看似方便分区，但容易把长期任务误建模成普通待办。

### 3. 日常任务必须隶属于长期任务，但不强制隶属于当前 TopOne

选择：日常任务创建时必须选择一个所属长期任务；若存在当前 TopOne，则创建表单默认选中当前 TopOne；如果当前没有 TopOne，用户仍可选择任意长期任务来创建日常任务。用户可以编辑和删除任何长期任务下的日常任务，但只有当前 TopOne 下的日常任务允许更新状态。

原因：需求明确每个长期目标拥有自己的日常任务池，但日常任务的维护不应被当前 TopOne 完全阻塞；限制状态更新到当前 TopOne 下，能保持专注推进语义。

备选：只允许当前 TopOne 下创建、编辑、删除日常任务。该方案过度限制用户整理任务库，也与用户确认的规则冲突。

### 4. 数量限制由服务层统一维护

选择：长期任务最多 3 个当前/暂停目标；每个长期任务下日常任务 `未开始 + 执行中` 最多 5 个、`执行中` 最多 2 个，这些规则由应用服务在创建和状态切换时统一检查。虽然只有当前 TopOne 下的日常任务可更新状态，但暂停任务库中的长期任务也使用同一活跃任务上限，避免任一长期任务的日常任务池过载。

原因：数量限制是业务规则，不应散落在视图按钮禁用逻辑中；服务层更容易测试，也能保障后续多入口一致。将日常任务上限绑定到具体长期任务，而不是只绑定当前 TopOne，能保证用户在整理暂停目标任务池时也不会创建过载计划。

备选：只在 UI 层禁用入口。该方案无法防止其他入口绕过限制。

### 5. 选择其他 TopOne 前必须先解绑当前 TopOne

选择：如果已经存在当前 TopOne，用户不能直接把另一个长期任务设为 TopOne；必须先发起解绑当前 TopOne。若仍处于锁定期内，解绑时要求填写提前更换理由，并执行 50 到 500 字的动态门槛规则；解绑完成后，用户才能选择其他长期任务作为新的 TopOne 并设置锁定时间。提前解绑会让 `earlySwitchCount` 加 1，并按 `50 × 2^earlySwitchCount` 计算下一次门槛，上限 500；完成长期任务时将 `earlySwitchCount` 减半后再计算门槛，用完成行为抵消一部分提前切换成本。

原因：用户确认“只有先解绑当前 TopOne，才能选择其他的长期任务作为 TopOne”；这能让更换目标成为显式动作，而不是在列表中无意识切换。用次数减半而非直接字数减半，可以让规则状态更容易持久化，也避免门槛在非 2 的幂次场景下出现不可解释的小数或取整争议。

备选：在选择新 TopOne 时自动解绑旧 TopOne。该方案交互更短，但会弱化锁定承诺与提前更换理由机制。

### 6. 任务页视觉组件必须按原型拆解实现

选择：任务页实现需要显式拆解为以下 SwiftUI 组件或等价结构：顶部 `TopOne` 导航栏、当前专注 Hero 卡片、日常任务行、左滑动作面板、暂停任务库分组、已达成里程碑区、右下角圆形 `+` 浮动按钮、专注锁定确认弹层、强制更换理由页。组件样式应参考原型色彩与层级：浅暖背景 `#fbf8fa`、深色主卡 `#091426` / `#1e293b`、金色强调 `#e9c400`、大圆角卡片、克制阴影、低密度留白。

原因：当前任务页不能退化成默认 `NavigationStack + Form + Menu` 的系统控件堆叠。TopOne 的任务页需要“当前 TopOne 是第一视觉锚点”，日常任务是推进层，暂停任务库默认弱化；这些都需要在组件结构与样式中被编码出来。

备选：继续使用系统默认列表和菜单快速承载功能。该方案虽然动态，但不符合 `docs/ui/README.md` 的“极简、克制、有精神性、有质感”要求，也不符合高保真原型的视觉组织。

### 7. 原型逻辑必须动态化，不得降级为行内菜单替代

选择：日常任务操作必须使用 SwiftUI `swipeActions` 或等价左滑交互表达 `更新 / 编辑 / 删除`；锁定 TopOne 必须使用专注选择/锁定确认弹层表达 `7 天 / 14 天 / 30 天 / 自定义`；提前更换必须进入强制更换理由页面或全屏弹层，展示理由输入、字数差值、规则说明与“再坚持一下 / 确定放弃”动作。

原因：这些交互在原型中承担产品语义：左滑是任务处理手势，锁定弹层表达承诺感，强制更换页表达更换成本。用普通行内 `Menu` 或系统确认框替代会丢失核心体验。

备选：所有操作都用 `Menu` / `confirmationDialog` 承载。该方案实现快，但已经被实际实现验证为偏离原型风格和逻辑。

## Risks / Trade-offs

- 本 change 覆盖范围较大 → 需要按领域模型、服务规则、UI 交互、测试分层实施，避免先做视觉堆叠。
- 锁定与提前切换理由涉及状态迁移 → 需要清晰区分正常设置、锁定期内提前切换、锁定期后切换。
- 自定义锁定时间需要输入校验 → 本轮将自定义锁定跨度限制为最长 180 天，避免无限制自由输入。
- 奖励池尚未实现 → 任务等级只保存与展示，不在本 change 中触发奖励校验或抽取。
