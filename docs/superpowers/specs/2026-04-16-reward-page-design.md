# 奖励页设计说明（2026-04-16）

> 2026-04-17 更新：本文档已按当前主分支实际实现对齐，描述的是当前已落地版本，而非最初设想中的目标态。

## 1. 背景与目标

当前 `TopOne` 已完成任务页动态闭环，后续功能优先规划奖励页与设置页。本轮先聚焦奖励页，并明确其第一目标不是复杂管理系统，而是为长期坚持提供**即时正反馈**。

奖励页第一版的核心体验是：

- 用户完成任务后获得积分
- 用户进入奖励页，主动选择奖励等级池
- 用户手动抽卡获得现实奖励
- 抽到的奖励进入个人库存，后续按需使用

本设计明确把奖励页定义为一个“激励闭环系统”，而不是单纯的奖励配置后台。

## 2. 设计目标

第一版奖励页需要满足以下目标：

- 以“抽卡正反馈”为首页核心叙事
- 奖励完全由用户自定义，而不是系统预置或自动生成
- 奖励分级沿用现有项目的 `S / A / B / C` 体系
- 积分获取与任务体系打通，但奖励消费仍保持独立页面与独立动作
- 把“奖励池定义”“抽到的奖励库存”“库存消费”三层语义严格区分
- 保持第一版范围克制，不引入概率运营、活动池、社交分享等复杂机制

## 3. 第一版产品定位

奖励页第一版采用“抽卡首页驱动型”方案。

产品结构上分为三个主要区域：

1. **奖励页首页**：负责积分查看、等级池切换、抽卡动作、奖励预览
2. **我的奖励**：负责查看已抽到奖励的库存并执行使用动作
3. **奖励管理**：负责用户自定义奖励的完整 CRUD

其中首页是底部导航中的“奖励”主页面，必须始终保持“抽卡优先”，管理和库存作为可达但次级的能力存在。

## 4. 信息架构

### 4.1 奖励页首页

奖励页首页是系统主入口，承担以下职责：

- 展示当前积分余额
- 通过分段切换器切换当前等级池
- 在当前等级池内执行抽卡
- 以循环轮播方式预览当前等级池奖励
- 提供“我的奖励”入口
- 提供统一的奖励管理入口

### 4.2 我的奖励

“我的奖励”是独立的二级页面，承担已获得奖励库存的浏览与消费，不承担奖励池管理能力。

该页面：

- 按 `S / A / B / C` 分组展示
- 同类奖励合并
- 显示当前奖励数量
- 数量为 `0` 时置灰显示
- 非零条目点击进入库存详情页

### 4.3 奖励管理

奖励管理是独立能力页，负责维护奖励池定义。

管理页的对象是“奖励定义”，不是“库存条目”。

每个奖励定义最少包含：

- 名称
- 图标
- 等级（`S / A / B / C`）
- 描述
- 抽取次数类型（有限 / 无限）
- 剩余次数（仅有限时）

## 5. 核心业务规则

### 5.1 奖励由用户定义

奖励池中的内容全部由用户手动创建和维护：

- 不提供系统内置奖励
- 不做 AI 自动生成奖励
- 不做模板化默认奖励池

### 5.2 奖励按 S / A / B / C 分级

所有奖励定义必须归属到一个等级池：

- `S`
- `A`
- `B`
- `C`

等级池既影响首页的抽卡入口，也影响积分消耗规则与视觉层级。

### 5.3 积分获取规则

积分不是自动持续增长，而是由任务完成事件触发。

第一版积分规则采用：

- **任务类型 + 等级** 双维度映射
- 长期任务与日常任务分别配置自己的给分规则
- `S / A / B / C` 任务等级在各自类型内再区分分值

积分发放时机严格限制为：

- **只有任务完成时才发放积分**

第一版明确不支持：

- 进入进行中即发分
- 按长期任务中间进度发分
- 过程行为发分

### 5.4 抽卡规则

用户在奖励页首页执行抽卡，流程为：

1. 选择某个等级池
2. 检查当前积分是否满足该等级池要求
3. 满足条件后点击抽卡
4. 系统扣除对应积分
5. 系统从该等级池中随机抽取一个可抽奖励
6. 将抽中结果加入“我的奖励”库存

第一版抽卡规则保持最小闭环：

- 同等级池内奖励默认等概率
- 不做权重、爆率、保底、连抽、活动池、限时池

### 5.5 奖励次数规则

每个奖励定义支持两种次数模式：

- **有限次数**
- **无限次数**

有限次数规则：

- 每抽中一次，剩余次数减一
- 当剩余次数为 `0` 时，该奖励在首页奖励预览区不再展示
- 但它仍保留在奖励管理中，供用户后续编辑或补充次数

无限次数规则：

- 始终可被抽取
- 不会因为抽卡而退出奖励池

第一版不支持：

- 自动恢复次数
- 按天/周/月重置次数

### 5.6 我的奖励库存规则

“我的奖励”页展示的是已经抽到的奖励库存，而不是奖励池定义。

库存规则如下：

- 同名奖励合并为一个库存条目
- 不记录获得时间
- 不做获得历史流水
- 不区分已使用 / 未使用两个独立分组
- 用“当前数量”表达持有状态
- 数量为 `0` 的条目仍保留，但置灰

### 5.7 使用奖励规则

用户不能在“我的奖励”列表页直接扣减数量，而必须进入详情页操作。

详情页提供一个统一动作：

- **使用奖励**

执行流程：

1. 用户点击“使用奖励”
2. 输入或选择本次使用数量
3. 系统校验不得超过当前持有数量
4. 弹窗二次确认
5. 用户确认后扣减库存数量

第一版不支持：

- 使用撤销
- 使用历史记录
- 使用时间追踪
- 多种消费动作类型

## 6. 页面设计

### 6.1 奖励页首页

首页信息层级如下：

#### 顶部资源状态区

展示：

- 当前积分余额
- 进入积分明细的入口

目的是让用户在进入奖励页的第一眼就知道：

- 自己现在有多少积分
- 可以直接进入积分变更记录与积分规则页

#### 等级池切换区

使用 `S / A / B / C` 分段切换器：

- 当前等级池有明确选中态
- 不同等级应有稀有感差异
- 切换后刷新下方详情区

#### 当前等级池详情区

由两部分组成：

1. **抽卡动作区**
   - 当前抽取消耗
   - 当前是否可抽
   - 主抽卡按钮
   - 无可抽奖励时的明确提示

2. **奖励预览区**
   - 单行循环轮播展示
   - 以图标为主，强化抽卡氛围
   - 当前实现按所选等级池展示奖励预览

#### 管理入口

首页当前提供统一管理入口。

### 6.2 奖励详情（池内定义详情）

当前版本暂未提供从首页奖励预览卡片进入的独立奖励定义详情页。

奖励定义的查看与维护主要在“奖励管理”页完成。

### 6.3 我的奖励

“我的奖励”页面按 `S / A / B / C` 分组，组内规则如下：

- 同类奖励合并
- 展示图标、名称、当前数量
- 当前数量为 `0` 的条目置灰

当前数量大于 `0` 的条目可点击进入奖励库存详情。

### 6.4 我的奖励详情（库存详情）

该页展示：

- 图标
- 名称
- 等级
- 描述
- 当前持有数量

并提供统一动作：

- 使用奖励

点击后：

- 用户选择本次使用数量
- 弹出二次确认
- 确认后扣减库存

### 6.5 奖励管理

奖励管理提供完整 CRUD：

- 新增奖励
- 编辑奖励
- 删除奖励

管理的是奖励池定义，不直接编辑用户库存。

## 7. 数据模型建议

第一版建议至少有三类核心数据概念：

### 7.1 用户奖励账户

用于存储：

- 当前积分余额

### 7.2 奖励定义

用于描述某个可抽取奖励：

- `id`
- `name`
- `icon`
- `rank` (`S / A / B / C`)
- `description`
- `availabilityType` (`finite` / `infinite`)
- `remainingCount`（有限时）

### 7.3 奖励库存条目

用于描述用户已经抽到的奖励：

- `rewardDefinitionId`
- `currentCount`

第一版不要求建立完整事件流水模型，因为：

- 不记录获得时间
- 不记录使用时间
- 不做审计级追踪

但抽卡扣积分与库存增加必须作为一个原子更新处理，避免状态不一致。

## 8. 异常与边界处理

### 8.1 积分不足

- 抽卡按钮禁用
- 不允许先抽后欠账
- 当前版本主要通过按钮状态表达不可抽，未额外显示差额提示

### 8.2 等级池为空

当某等级池没有任何可抽奖励时：

- 抽卡按钮禁用
- 给出明确提示
- 引导用户进入奖励管理页补充该等级奖励

### 8.3 使用数量非法

若用户在库存详情页输入的使用数量：

- 小于等于 `0`
- 大于当前持有数量

系统应拒绝提交，并给出明确提示。

### 8.4 库存扣减后归零

当使用奖励导致某条库存数量归零时：

- 保留该条目
- 以置灰态显示

### 8.5 有限奖励抽空

当奖励定义剩余次数归零时：

- 首页不再展示该奖励
- 奖励管理页中保留该定义
- 用户可后续编辑补充次数或调整配置

## 9. UI / 产品风格约束

奖励页必须与当前 `TopOne` 产品风格保持同源，而不是演化为独立的游戏化视觉系统。

### 9.1 风格继承原则

奖励页应延续当前任务页已经建立的核心产品语言：

- 浅暖背景
- 深色核心卡片
- 金色强调
- 低密度留白
- 大圆角
- 克制阴影

奖励页可以在此基础上增强情绪张力，但不能脱离当前产品的整体气质。

### 9.2 风格增强优先级

奖励页的视觉增强优先级明确为：

1. **抽卡揭晓体验**
2. **奖励卡片的收藏感 / 稀有感**
3. **首页整体氛围**

这意味着风格强化应集中在关键体验瞬间，而不是让整个页面长期处于高噪声状态。

### 9.3 色彩策略

奖励页应采用“同一主配色下做轻微扩展”的方案：

- 主色体系延续任务页现有配色
- 允许局部加入少量更有光泽感、揭晓感的高光色
- 高光色只服务于抽卡揭晓、奖励确认、稀有感表达
- 不将 `S / A / B / C` 设计为四套彼此割裂的强分色系统

### 9.4 文案气质

奖励页文案应采用“两者结合”的策略：

- 整体保持克制、安静、带精神性的表达
- 在“抽卡成功”“获得奖励”“使用奖励”等关键节点，允许更明确表达奖励感与兑现感

也就是说，奖励页不是轻浮游戏化口吻，而是在当前产品语言上增加更有收获感的情绪反馈。

### 9.5 动效原则

奖励页动效应强调：

- 揭晓
- 聚焦
- 确认

不应使用：

- 高频闪烁
- 夸张粒子轰炸
- 持续强刺激的游戏厅风格反馈

动效的目标是让用户感受到“奖励被认真交付”，而不是制造廉价兴奋。

### 9.6 首页信息密度约束

奖励页首页必须继续遵守“一个主动作优先”的原则：

- 首页主动作是抽卡
- 管理与库存能力必须可达，但不应抢占首页叙事中心
- 不把首页堆成后台式控制台或信息面板

## 10. 第一版明确不做

第一版奖励页暂不实现以下能力：

- 权重概率系统
- 爆率调节
- 保底机制
- 连抽
- 活动池、限时池
- 社交分享
- 系统预置奖励
- AI 自动生成奖励
- 自动恢复次数
- 获得时间记录
- 使用时间记录
- 使用历史流水
- 使用撤销

## 11. 推荐实施顺序

建议后续实现顺序为：

1. 先建立奖励定义与库存模型
2. 再建立积分账户与任务完成发分映射
3. 实现奖励页首页的等级池切换与可抽状态
4. 接入抽卡动作与库存增加
5. 实现“我的奖励”与库存消费流程
6. 最后补齐奖励管理的完整 CRUD 与页面细节

这样可以尽快形成最小闭环：

**任务完成得积分 → 进入奖励页抽卡 → 奖励入库存 → 手动使用奖励**

## 12. 设计结论

奖励页第一版应该被实现为一个“抽卡首页驱动的个人激励系统”。

它的核心不是复杂游戏化运营，也不是后台式奖励配置，而是围绕用户自定义现实奖励，建立一个克制但完整的正反馈循环：

- 用任务完成换积分
- 用积分换抽卡机会
- 用抽卡获得真实奖励
- 用库存管理把奖励真正消耗出去

只要这个闭环成立，奖励页就已经能与当前任务页形成明确分工：

- 任务页负责“专注推进”
- 奖励页负责“正反馈兑现”

这也是第一版最应该守住的产品中心。
